What does internationalization (i18n) even mean?

  • Translating application messages... that's it right? No!
  • Localizing for each country and region should be easy
  • This is more than just changing locale, currency, date etc.
  • i18n !== translating text from one locale to another
  • Writing app in a way that localizing it for each country and region is easy
  • Integrated process with translators

Holy...!

  • What if I'm not prepared for such a progress?
  • Also what about legacy apps?
  • Do I have to start all over again?

A good solution lets you localize your apps in whatever shape they are.

  • It shouldn't matter how old your web app is
  • We should be able to make incremential changes
  • That's the vision for core i18n support in Angular
  • Let's take a look at what the i18n process could look like
  • Developer writes HTML templates
  • Templates can be directive or router view tpls in Angular world
  • Messages get extracted in locale they are written in
  • We provide tools for this, but you can integrate them into your toolchain as well
  • Somehow bundled and transformed to file format that translator needs
  • Message bundles are given to translator
  • Translator translate bundles to other locale
  • Translator returns translated bundles in new locale
  • Developer generates new templates from given bundles
  • Again, we provide tools for this
  • As we can see, this process is iterative
  • We might add messages to the template over time etc.
  • How often do we translate?

Bug report

Your UI has text that's cut off. The spacing is all weird and some messages are still in English.

Solution: Pseudotranslation

Pseudotranslation provides INSTANT translation into gibberish. Such gibberish will have features such as:

  • Sentences size doubled by repeating words
  • Characters munged in some way such as being accented
  • Brackets on either end to easily see text that's cut off

Example

Message:

Enter your name

Pseudotranslated Message:

[Enter Ën̈ẗër̈ your ÿöür̈ name n̈äm̈ë]
  • Catch text that's cut off
  • Catch issues with spacing
  • Catch messages that came from Javascript or were others not extracted
  • Make it part of your workflow (pseudotranslate on save.)

Internationalization (i18n) is the process of developing products in such a way that they can be localized for languages and cultures easily.

Localization (l10n) is the process of adapting applications and text to enable their usability in a particular cultural or linguistic market.

I18n support in the Angular world (today)

  • ngLocale - Static Angular extension used by built-in components (currency, date, ...)
  • angular-gettext - gettext utilities for Angular
  • angular-translate - i18n support through translation IDs
  • ...

Problems with current solutions

May or may not apply to all of them.

  • No way to annotate translations with meta information for translator
  • ID abstraction instead of language as default
  • Poor pluralization support
  • Works only on the client
  • Many more! (Angular and i18n: A new world)

We want a first-class i18n story for Angular v1 and v2!

Main Goals

  • Tool support - Message extraction, Format Transformer, Application Generator, ...)
  • Server/Client-side Support - Static Mode, Dynamic Mode
  • Gender and Pluralization - Using ICU MessageFormat syntax
  • Performance - It's a feature
  • ...

Pluralization and Gender

Pluralization and Gender?

Pluralization:

You have 1 new message.
You have 2 new messages.

Gender:

She has sent you an email.
They have sent you an email.

Pluralization and Gender?

Both can be combined:

He found 1 result in 2 categories.
She found 1 result in 2 categories.
He found 2 results in 1 category.
She found 2 results in 1 category.
They found 2 results in 2 categories.

First-class MessageFormat integration

Some reasons why we decided to do so:

  • Standard syntax, translators are familiar with it
  • Syntax is compact and easy to used in Angular templates
  • Can be used inside attributes
  • Supports nesting of plurals and gender selections

MessagFormat Syntax

{GENDER, select,
    male {He found one category.}
  female {She found one category.}
   other {They found one category.}
}

Gender selection

MessagFormat Syntax

You have {NUM_RESULTS, plural,
    one {1 message.}
  other {# messages.}
}

Plural selection

MessagFormat Syntax

{GENDER, select,
    male {He}
  female {She}
   other {They}
} found {NUM_RESULTS, plural,
            one {1 result}
          other {# results}
        } in {NUM_CATEGORIES, plural,
                  one {1 category}
                other {# categories}
             }.

Angular specific concerns

  • MF's argument syntax is restricted to Pattern_Syntax (a.k.a. no expressions allowed)
  • Angular uses double curlies for interpolation, conflicts with MF's syntax
  • A curly inside Angular expression represents an object, while in MF it causes mode switch

MessageFormat in Angular

{{recipients.length, plural,
  =0 { You, {{user.name}}, have not sent any messages. }
  =1 { You, {{user.name}}, have sent a message. }
  other { You have sent {{recipients.length}} people messages. }
}}

MessageFormat syntax can be used inside Angular expressions.

MessageFormat in Angular

{{recipient.gender, select,
  female { You, {{user.name}}, have sent her messages. }
  male { You, {{user.name}}, have sent him messages. }
  other { You, {{user.name}}, have sent them messages. }
}}

MessageFormat syntax can be used inside Angular expressions.

MessageFormat in Angular

{{recipients.length, plural,
  =0 {You, {{user.name}}, have not sent any messages.}
  =1 {recipients[0].gender, select,
        female {You, {{user.name}}, sent her a message.}
        male {You, {{user.name}}, sent him a message.}
        other {You, {{user.name}}, sent them a message.}
     }
  other {You sent {{recipients.length}} people messages.}
}}
